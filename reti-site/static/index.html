Methodology and Scope

In Karsten Müller and Frank Lamprecht's classic work Fundamental Chess Endings, they give a table showing how often each ending from selected chapters occurs.1 A simplified version of this table is available on Wikipedia.2 To produce the original table in 2001, Müller and Lamprecht used ChessBase and that year's Mega Database, containing about 1.7 million games. Their analysis used Fritz 6 and Nimzo 8, with five-man tablebases for positions that were already solved.

The chess software landscape looks very different in 2026. Stockfish 18 is one of the strongest available engines, and it is free and open-source software. FOSS chess tools are now ordinary rather than exceptional. Large collections of games are also easier to obtain through aggregators such as Lumbra's Gigabase,3 whose own source list includes material from Lichess,4 PGNMentor,5 TWIC,6 and many other sources. Five-man tablebases are still useful, but complete seven-piece tablebases are now standard, and selected larger tablebase projects exist for more specialised cases.

I wanted to see how far the original Müller-Lamprecht analysis could be reproduced and extended using FOSS tools. I downloaded roughly 17 million games from Lumbra's Gigabase: about 10 million over-the-board games and about 7 million high-level online games. This is roughly an order of magnitude larger than the database used for the original table.

The next problem is locating and categorising the endings. This project uses Chess Query Language, originally developed by Gady Costeff and Lewis Stiller,7 via Robert Gamble's CQLi implementation.8 The CQL scripts used to generate the annotated PGN files are in the Reti GitHub repository.9 CQL is a powerful language for filtering chess positions; the filters used here are deliberately simple material-pattern queries.

I was also interested in how often tablebase evaluations line up with the final result of the game. For that reason, every qualifying first marker position with five pieces or fewer is evaluated with Syzygy WDL. The current build does not run Stockfish evaluations: the engine comparison is left to Lichess analysis when you click through to a position. For each game-ending pair, the tablebase statistics use the first matching position, not every matching position in the game. For example, the row "8.2a Rook + Bishop vs Rook without pawns" can show a higher practical win rate than the tablebase result alone would suggest, which is plausible because this ending is easy for humans to mishandle. I was also surprised by the practical win rate for bishop and knight versus king: it is theoretically won, but still non-trivial over the board. Switching the corpus from OTB to Online gives a quick way to compare longer and shorter practical time-control environments.

I am not reproducing the original published table in full here. The public Wikipedia summary is linked for context, and the dashboard focuses on my recomputed Gigabase statistics. The broad agreement with Müller and Lamprecht is interesting in itself. One hypothesis is that opening choice influences which pieces are exchanged, which in turn changes the distribution of endings that arise; that remains a useful direction for later analysis, but it is not part of this published dashboard.

Clicking on an ending expands additional statistics and sampled games. At most 32 games are sampled for each selected corpus, threshold, and ending; if fewer qualify, all available games are shown. Clicking on a board opens the position on a Lichess analysis board, so you can inspect it with current engines and tablebases. The fact that this is now possible with open tools and public data is the main point of the project.
